Vapor is built on [SwiftNIO](https://github.com/apple/swift-nio), an event-driven, non-blocking
network framework. Swift's structured concurrency (`async`/`await`) is the recommended way to write
Vapor route handlers and middleware today.

## Route handlers with `async`/`await`

Every HTTP method helper on `RoutesBuilder` accepts an `async throws` closure:

```swift
app.get("hello") { req async throws -> String in
    return "Hello, world!"
}
```

The closure signature is `@Sendable @escaping (Request) async throws -> Response` where `Response`
must conform to `AsyncResponseEncodable`. Return types that are `String`, `Codable`, `Response`, or
custom types conforming to `Content` all work out of the box.

```swift
struct User: Content {
    var id: UUID
    var name: String
}

app.get("users", ":id") { req async throws -> User in
    let id = try req.parameters.require("id", as: UUID.self)
    // fetch from database …
    return User(id: id, name: "Alice")
}
```

## Sendable requirements

Vapor marks route closures `@Sendable`. Any value captured in a route closure must itself be
`Sendable`. This is enforced by the Swift compiler and aligns with Swift 6 strict concurrency
checking.

```swift
// Fine — CounterService is Sendable
let counter = CounterService()

app.get("count") { req async throws -> String in
    counter.increment()
    return "\(counter.value())"
}
```

<Warning>
Capturing non-`Sendable` types in route closures produces a compiler error under Swift 6 strict
concurrency. Use actors or `NIOLockedValueBox` to protect mutable state.
</Warning>

## Making HTTP client requests

`Request` exposes a non-blocking HTTP client via `req.client`. All methods are `async throws`:

```swift
app.get("proxy") { req async throws -> String in
    let response = try await req.client.get("https://example.com/api/data")
    guard let body = response.body else {
        throw Abort(.badGateway)
    }
    return String(buffer: body)
}
```

You can also send JSON:

```swift
struct Payload: Content {
    var message: String
}

app.post("notify") { req async throws -> HTTPStatus in
    let payload = Payload(message: "Hello from Vapor")
    let response = try await req.client.post(
        "https://hooks.example.com/incoming",
        content: payload
    )
    guard response.status == .ok else {
        throw Abort(.badGateway)
    }
    return .noContent
}
```

Available methods on `Client`: `get`, `post`, `patch`, `put`, `delete`, `send`.

## Decoding request content

Use `req.content.decode(_:)` to parse the request body:

```swift
struct CreateUser: Content {
    var name: String
    var email: String
}

app.post("users") { req async throws -> HTTPStatus in
    let dto = try req.content.decode(CreateUser.self)
    // persist dto …
    return .created
}
```

## Structured concurrency inside handlers

Route handlers are async contexts, so you can use Swift's structured concurrency APIs directly:

```swift
app.get("parallel") { req async throws -> [String] in
    async let first = req.client.get("https://api.example.com/one")
    async let second = req.client.get("https://api.example.com/two")

    let (r1, r2) = try await (first, second)
    return [r1.description, r2.description]
}
```

## Running blocking work off the event loop

If you need to call a synchronous, potentially blocking API (e.g. a legacy library), use
`app.threadPool` via `NIOThreadPool` and `NonBlockingFileIO`, or dispatch the work to a detached
`Task`:

```swift
app.get("blocking") { req async throws -> String in
    // Offload to a separate thread so the event loop isn't blocked.
    return try await Task.detached(priority: .userInitiated) {
        // synchronous, potentially blocking work here
        return expensiveSynchronousComputation()
    }.value
}
```

Alternatively, use Vapor's `fileio` for file system access without blocking:

```swift
app.get("file") { req async throws -> Response in
    try await req.fileio.streamFile(at: req.application.directory.publicDirectory + "image.png")
}
```

## `EventLoop` and `EventLoopFuture` (legacy)

Before Swift concurrency was available, Vapor used `EventLoopFuture<T>` (a promise/future type from
SwiftNIO) throughout. You may still encounter it in older code or third-party packages.

<Tabs>
  <Tab title="async/await (recommended)">
    ```swift
    app.get("item") { req async throws -> Item in
        try await database.find(Item.self, id: 42)
    }
    ```
  </Tab>
  <Tab title="EventLoopFuture (legacy)">
    ```swift
    app.get("item") { req -> EventLoopFuture<Item> in
        database.find(Item.self, id: 42)
    }
    ```
  </Tab>
</Tabs>

You can bridge futures to async/await with `.get()`:

```swift
let value = try await someEventLoopFuture.get()
```

And convert async work back to a future when needed:

```swift
let future = req.eventLoop.makeFutureWithTask {
    try await someAsyncFunction()
}
```

## Async middleware

Implement `AsyncMiddleware` to intercept requests and responses in async code:

```swift
struct LoggingMiddleware: AsyncMiddleware {
    func respond(
        to request: Request,
        chainingTo next: AsyncResponder
    ) async throws -> Response {
        request.logger.info("→ \(request.method) \(request.url.path)")
        let response = try await next.respond(to: request)
        request.logger.info("← \(response.status.code)")
        return response
    }
}

// Register globally
app.middleware.use(LoggingMiddleware())

// Or on a route group
let protected = app.grouped(LoggingMiddleware())
protected.get("dashboard") { req async throws in /* … */ }
```

## The `Application` entry point

When you create an application with `Application.make(_:_:)`, the async entry point looks like this:

```swift
@main
struct Entrypoint {
    static func main() async throws {
        var env = try Environment.detect()
        try LoggingSystem.bootstrap(from: &env)

        let app = try await Application.make(env)
        do {
            try configure(app)
            try await app.execute()
            try await app.asyncShutdown()
        } catch {
            try? await app.asyncShutdown()
            throw error
        }
    }
}
```

`app.execute()` starts the server and suspends until a shutdown signal (`SIGINT` or `SIGTERM`)
is received, then resumes and calls `app.asyncShutdown()` to clean up.

## Actors as application-level state

Swift actors are a natural fit for shared mutable state across concurrent requests:

```swift
actor MemoryCache {
    private var storage: [String: String] = [:]

    func get(_ key: String) -> String? { storage[key] }
    func set(_ key: String, to value: String?) { storage[key] = value }
}

let cache = MemoryCache()

app.get("cache", ":key") { req async throws -> String in
    let key = try req.parameters.require("key")
    return await cache.get(key) ?? "nil"
}
```

<Note>
Actors are `Sendable`, so they can be safely captured in `@Sendable` route closures.
</Note>
